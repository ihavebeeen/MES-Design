# 모바일 컴포넌트 체크리스트

이 문서는 피그마 "컴포넌트 라이브러리" 파일의 **MES Mobile** 페이지(모바일/태블릿 전용 디자인 시스템)를 대조해 정리한 목록이다. [UI 컴포넌트 총망라 체크리스트](./UI_컴포넌트_총망라_체크리스트.md)(데스크탑 기준)와 동일한 레이어/상태/티어 형식을 따른다.

**출처**: [Figma — 컴포넌트 라이브러리 / MES Mobile](https://www.figma.com/design/qEAQNTqXf76bYydyupti2V/%EC%BB%B4%ED%8F%AC%EB%84%8C%ED%8A%B8-%EB%9D%BC%EC%9D%B4%EB%B8%8C%EB%9F%AC%EB%A6%AC?node-id=163-8929)

**주요 발견**: 데스크탑 파일과 달리, 모바일/태블릿용 디자인 시스템은 이미 상당 부분 구축되어 있다. Foundation(토큰), 주요 아톰, 그리고 11개 화면 템플릿까지 존재. 아래 표의 `상태`는 대부분 `있음`이며, 목적은 "무엇이 이미 있는지"를 데스크탑과 같은 형식으로 문서화하는 것이다.

## 사용법

레이어/티어 정의는 데스크탑 체크리스트와 동일. `상태` 컬럼: `있음`/`부분`/`없음`. `티어`는 데스크탑 체크리스트의 우선순위 기준(1~5)을 그대로 적용하되, 모바일 화면 자체의 존재 여부가 아니라 "웹 코드화가 끝난 뒤 모바일을 어느 순서로 다룰지"를 의미한다.

**결론**: 기존 확인 항목(토큰·OS 레벨·아톰·MES 특화·템플릿 약 30개) + 이번 표준 대조로 새로 식별된 항목(중복 제외 약 25개)을 합치면 모바일에서 실제로 필요한 컴포넌트 범위의 약 90%+ 를 총망라했다고 볼 수 있다. 데스크탑과 동일하게 **티어 1~3까지가 실행 목표**이며, 티어 2에 FAB·Chip·Card·Badge·Divider·Progress Indicator·Date/Time Picker·Checkbox·Action Sheet·Swipe Action·Empty/Error State가 새로 포함된 점이 데스크탑 우선순위와의 가장 큰 차이다.

## 9. 데스크탑 대비 공백 (모바일에서 아직 확인 안 된 것)

데스크탑 체크리스트에 있는 항목 중, 이번 모바일 파일 조사에서 확인되지 않은 것들. 실제로 없는 것인지 단순히 이 프레임 범위 밖에 있는 것인지는 추가 확인 필요.

| 항목 | 비고 |
|---|---|
| Modal / Dialog (중앙 팝업) | 모바일에선 보통 풀스크린 또는 Bottom Sheet로 대체되는 경우가 많음 — 별도 패턴 확인 필요 |
| Data Grid / Table (복잡한 표) | 모바일 화면 특성상 리스트 카드형으로 대체됐을 가능성 — "리스트 화면" 템플릿 내부 구조 추가 확인 필요 |
| Toast / Snackbar | 확인 안 됨 |
| Pull to Refresh, Swipe Action | 확인 안 됨 (위 1번 표에도 기재) |
| Empty State / Error State | 확인 안 됨 |

## 다음 단계

1. 데스크탑(웹) 티어 1~3 코드화가 우선이며, 모바일은 그 이후 별도 트랙으로 진행.
2. 모바일은 이미 Foundation + 주요 아톰 + 11개 화면 템플릿까지 디자인이 갖춰져 있어, 웹보다 오히려 **코드화 시작 조건이 더 좋음** — 실제 착수 시점은 사업 우선순위에 따라 결정.
3. 8번(표준 컴포넌트 대조)에서 새로 식별된 티어 2 항목(FAB, Chip, Card, Badge, Divider, Progress Indicator, Date/Time Picker, Checkbox, Action Sheet, Swipe Action, Empty/Error State 등)은 웹 티어 1~3 작업과 함께, 또는 그 직후에 디자인부터 새로 만들어야 함 — 현재 피그마엔 없는 항목이라 코드화 이전에 디자인 단계가 선행되어야 함.
4. 9번 "공백" 항목은 코드화 직전에 해당 프레임을 다시 훑어 확인.
5. 색상 토큰(Warning/Success)은 데스크탑 파일에도 동일하게 있을 가능성이 높으므로, 데스크탑 Color 프레임 재확인 시 함께 반영.
