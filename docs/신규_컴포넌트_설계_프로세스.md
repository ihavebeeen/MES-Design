# 신규 컴포넌트 설계 프로세스 (SOP)

기존 전체 파이프라인의 4번째 단계 "새 컴포넌트 설계"를, 어떤 컴포넌트가 오더라도(액션류든 오버레이류든 MES 특화든) 동일하게 적용할 수 있도록 일반화한 절차다. [UI 컴포넌트 총망라 체크리스트](./UI_컴포넌트_총망라_체크리스트.md), [모바일 컴포넌트 체크리스트](./모바일_컴포넌트_체크리스트.md)에서 `상태=없음`으로 표시된 모든 항목에 이 절차를 동일하게 적용한다.

**전체 파이프라인에서의 위치**: 1) 기존 컴포넌트 실측 → 2) 패턴으로 묶어 토큰표 작성 → 3) 기존(티어1) 컴포넌트 코드화 → **4) 신규 컴포넌트 설계 (본 문서)** → 5) 티어 순서대로 반복

---

## Step A. 컴포넌트 패밀리 분류

새로 만들 컴포넌트를 아래 5개 패밀리 중 하나로 분류한다. 패밀리가 다르면 적용할 토큰 프로파일(radius/shadow/outline 경향)이 달라지기 때문에, 설계의 출발점은 항상 이 분류다.

| 패밀리 | 정의 | 이미 있는 예시(참고용) | 신규 항목 예시 |
|---|---|---|---|
| 액션류 | 클릭해서 즉시 동작을 일으키는 요소 | Button, 검사 버튼(○△✕) | FAB, Icon Button, Segmented Control, Chip(선택형) |
| 입력류 | 값을 입력·선택받는 요소 | Text Input, Selectbox, Toggle | Checkbox, Date/Time Picker, Slider, Stepper |
| 표시·컨테인먼트류 | 정보를 담아 보여주는 요소 | Badge, Chip/Tag, Stat Card | Card, Divider, Progress Indicator, Skeleton, Avatar |
| 오버레이류 | 기존 화면 위에 뜨는 요소 | Center Modal, Side Slide Panel, Toast(Figma 확인분) | Action Sheet, Bottom Sheet, Popover, Context Menu, Snackbar |
| 내비게이션류 | 화면 간 이동·구조를 담당하는 요소 | Sidebar, Tabs, Header, Bottom Tab Bar | Navigation Drawer, Breadcrumb |

> 하나의 컴포넌트가 두 패밀리에 걸치는 경우(예: Combobox = 입력류+오버레이류)는 주기능 기준으로 1차 분류하고, 나머지 패밀리의 토큰 규칙도 함께 확인한다.

## Step B. 가장 가까운 기존 컴포넌트 선정 (Nearest Neighbor)

같은 패밀리 안에서, 구조나 목적이 가장 비슷한 **이미 있는(티어1)** 컴포넌트를 1~2개 고른다. 완전히 새로 그리지 않고 이 컴포넌트를 포크·확장하는 방식으로 접근한다.

| 새 컴포넌트 | 가장 가까운 기존 컴포넌트 | 확장 방식 |
|---|---|---|
| Card | Stat/KPI Card (있음) | 여백·그림자 구조는 그대로, 내용 슬롯만 범용화 |
| Action Sheet | Center Modal / Side Slide Panel (있음) | 위치를 하단 고정으로, 옵션 리스트 형태로 단순화 |
| Badge (알림 배지) | Chip/Tag, 검사 버튼의 상태 배지 (있음) | 크기를 더 작게, 숫자 전용 레이아웃 추가 |
| FAB | Button 매트릭스의 fill-main 스타일 (있음) | 원형 shape + elevation(그림자) 추가 |
| Navigation Drawer | Sidebar (있음) | 데스크탑 고정형 → 모바일 슬라이드 오버레이형으로 전환 |

이 표는 예시이며, 실제 설계 시점마다 그 컴포넌트에 맞는 참고 대상을 다시 선정한다. 원칙은 "완전 백지에서 시작하지 않는다"이다.

## Step C. 토큰 매핑 규칙 적용

Step 2(패턴으로 묶어 토큰표 작성)에서 나올 "패밀리별 토큰 프로파일"을 그대로 대입한다.

| 토큰 종류 | 규칙 |
|---|---|
| Corner Radius | 소속 패밀리의 대표 radius 값을 그대로 사용 (예: 액션류는 버튼 radius, 오버레이류는 모달 radius) |
| Shadow/Elevation | 오버레이류·FAB류만 그림자 적용, 액션류(버튼)·입력류는 기본적으로 무그림자 원칙 유지 (기존 패턴과 동일하게) |
| Outline/Border | 입력류는 기존 Input/Selectbox의 outline 색상·두께 그대로, 표시류는 필요 시에만 |
| Color | 기존 Color Palette(Gray/Primary/Danger/Warning/Success)에서만 선택 — 새 hex 임의 추가 금지 |
| Spacing/Padding | 기존에 확인된 spacing 스케일 중에서 선택 |
| Typography | 기존 Typography Scale(Pretendard, 정해진 사이즈 스텝) 중에서 선택 |

**절대 규칙**: 토큰표에 없는 값(임의의 radius, 임의의 hex, 임의의 폰트 사이즈)은 새 컴포넌트에 절대 쓰지 않는다. 필요하면 토큰표 자체를 먼저 확장하고 나서 적용한다.

### Step C-1. 실측 파일럿 결과 (2026-07-16, `get_design_context` 9개 노드 / 5개 컴포넌트)

**방법론 관련 수정사항**: 최초 1차 시도(4개)는 메타데이터에서 이름으로 grep해 아무 노드나 잡아 실측했는데, 나중에 확인해보니 Button/Input으로 잡았던 노드가 컴포넌트 정의부가 아니라 실제 화면("판매목표(월) 리스트") 안에 쓰인 인스턴스였다. 인스턴스는 오버라이드 가능성이 있어 신뢰도가 낮다. 그래서 파일 구조를 먼저 지도화해 진짜 정의부("Foundation (Atoms)" 구역, 노드 `163:5987` 주변)를 특정하고, 거기서 컴포넌트당 2개 이상 변형(사이즈별)을 교차 확인하는 방식으로 다시 진행했다. Badge·Center Modal은 처음부터 이 Foundation 구역 안에 있던 노드라 재검증 없이 유효.

| 컴포넌트 | 패밀리 | 변형 | Radius | Height/Size | Padding/Gap | Color | 폰트 | 확인 상태 |
|---|---|---|---|---|---|---|---|---|
| Button (fill-main) | 액션류 | size=basic | 4px | 40px | 좌우 16px, 아이콘-텍스트 gap 6px | bg `#1573FB`, text `#FFFFFF` | Pretendard SemiBold 15px | 정의부 노드 확인 |
| Button (fill-main) | 액션류 | size=small | 4px | 32px | 좌우 10px, gap 6px | bg `#1573FB`, text `#FFFFFF` | Pretendard SemiBold 14px | 정의부 노드 확인 |
| Input Textbox | 입력류 | size=Default | **4px** | 32px | 좌우 10px | bg `#FFFFFF`, border 1px `#E4E7EE`, placeholder `#99A3B8` | Pretendard Regular 14px | 정의부 노드 확인 |
| Input Textbox | 입력류 | size=small | **3px** | 22px | 좌우 6px | bg `#FFFFFF`, border 1px `#E4E7EE`, placeholder `#99A3B8` | Pretendard Regular 13px | 정의부 노드 확인 |
| Input Textbox (검색 아이콘형) | 입력류 | 아이콘 포함, 화면 실사용 인스턴스 | 4px | 32px | 좌우 12px | bg `#EFF2F8`(플레인형과 다름 — 아이콘형은 배경 있음) | Pretendard Regular 14px | 인스턴스(참고용) |
| Badge | 표시·컨테인먼트류 | 기본 | 4px(`radius/small1`) | — | 좌우 8px(`padding/3`), 상하 2px(`padding/1`) | bg `#F04452`(System Red 60) | Pretendard Bold 20px | 정의부 노드 확인 |
| Center Modal (삭제 모달) | 오버레이류 | 기본 | 컨테이너 16px, 내부 카드 12px, 버튼 4px | — | — | — | — | 정의부 노드 확인 |
| 모달 그림자 | 오버레이류 | 기본 | — | — | — | — | `0px 0px 24px rgba(0,0,0,.2)`(컨테이너), `0px 4px 16px rgba(0,0,0,.2)`(내부 카드) | 정의부 노드 확인 |
| Page Top Tab Item | 내비게이션류 | Default | 내부요소 4px/3px 혼재 | 38px | 좌우 12px, gap 10px/4px | bg `#000000`, border-r 1px `#373F57`, text `#FFFFFF` | Pretendard Regular 13px | 정의부 노드 확인 |

**핵심 수정 — "radius 4px 공용 토큰" 결론은 틀렸다**: 1차 시도에서 "액션류·입력류·표시류가 전부 4px로 통일"이라고 결론 내렸는데, Input의 small 사이즈를 교차 확인해보니 **3px**였다. Button은 basic/small 둘 다 4px로 동일했지만, Input은 사이즈에 따라 radius 자체가 달라진다 — 컴포넌트마다 radius가 사이즈에 연동되는지 여부가 다르므로, "기본 radius=4px, small radius=3px" 같은 단일 규칱을 전체에 적용하면 안 되고 **컴포넌트별로 각각 확인해야 한다.**

**주의(확인 필요, 19번 섹션급 오픈 이슈)**:
1. Badge에서 발견된 `padding/3`(8px), `padding/1`(2px), `radius/small1`(4px)은 파일 로컬 변수이고, `search_design_system`으로는 안 잡힌다 — 대신 "FMS Make Metrics"라는 별도 변수 컬렉션(`radius/sm`, `radius/md`, `radius/lg`, 값 미확인)과 "gray-900: #191F28" 같은 "동진MES DS/" 접두어 없는 색상도 발견됨 — 둘 다 실사용 중인 "동진MES DS" 체계와는 별개로 보이는 미사용 보일러플레이트일 가능성이 높다. 코드화 시 착오로 이 변수들을 참조하지 않도록 주의.
2. **Gray 60 색상값 불일치**: 체크리스트 0-1번 표엔 `#96A0B5`로 기재돼 있는데, Input Textbox 두 개 정의 노드(Default/small) 모두에서 독립적으로 `#99A3B8`이 나왔다(2회 교차확인) — 체크리스트 값이 오기재였을 가능성이 크다. 전체 팔레트 재실측 필요.
3. Radius가 사이즈별로 달라지는 패턴(Input 4px→3px, Tab item 내부요소 4px/3px 혼재)이 Button 이외 컴포넌트에도 있는지 — 코드화 직전 해당 컴포넌트마다 개별 확인 필요.

## Step D. 상태(State) 세트 결정

새 컴포넌트가 속한 패밀리의 기존 컴포넌트들이 공통으로 갖는 상태 세트를 그대로 요구사항으로 가져온다.

| 패밀리 | 최소 요구 상태 세트 |
|---|---|
| 액션류 | default / hover / press(active) / disabled (+ 선택형이면 selected) |
| 입력류 | 입력전 / focus / 입력완료 / readonly / disable / error |
| 표시·컨테인먼트류 | default (+ 상태값을 담는 경우 색상 변형: success/warning/danger) |
| 오버레이류 | 진입(open) / 종료(close) (+ 애니메이션 방향 명시: 하단에서 슬라이드 등) |
| 내비게이션류 | default / active(현재 위치) / disabled |

빠진 상태가 하나라도 있으면 "부분"으로 남기고, 전체 상태를 다 정의해야 "있음"으로 체크리스트에 반영한다.

## Step E. 사이즈 체계 적용

새 컴포넌트도 기존에 이미 쓰이는 사이즈 스텝(예: default / small / Xsmall, 모바일은 default / small)을 그대로 재사용한다. 새로운 사이즈 단계가 정말 필요한 경우에만 예외로 추가하고, 그 사유를 문서에 남긴다.

## Step F. MES 도메인 요소 반영

MES 특화 신규 컴포넌트(예: 알람 관련, 설비 관련 신규 위젯)는 위 A~E를 적용한 뒤, 추가로 아래를 확인한다.

- 상태 색상이 Danger(불량/위험) / Warning(주의) / Success(정상/합격) 의미 체계와 일치하는가 (검사 버튼 ○△✕ 패턴과 동일한 원칙)
- 라벨·용어가 기존 화면(작업지시, 품질검사, 설비보전 등)과 동일한 한글 용어를 쓰는가
- 아이콘은 기존 Icon Set(데스크탑/모바일 각각)에서 우선 재사용하고, 없을 때만 신규 아이콘을 만든다
