# CLAUDE.md — 이 저장소에서 작업할 때 지켜야 할 규칙

이 파일은 Claude(코드/디자인 작업 담당)가 이 레포에서 새 세션을 시작할 때마다 가장 먼저 읽어야 하는 파일이다. PC/Mac 등 어떤 기기에서 열든 동일하게 적용한다.

## 매 세션 시작 시 반드시 할 것

1. `git pull origin main` 먼저 실행 — 다른 기기(PC/Mac)에서 작업한 최신 내용을 받아온다.
2. `docs/` 폴더의 체크리스트 문서들과 Notion 페이지(DESIGN COMPONENT, MOBILE COMPONENT)를 확인해서 현재 진행 상태를 파악한다.
3. 사용자에게 "이어서 할 작업"을 다시 설명하지 않고, 파악한 상태 그대로 이어서 진행한다.

## 작업 마무리 시 반드시 할 것

1. 변경사항을 의미 단위로 `git commit` 한다 (커밋 메시지는 한글로, 무엇을 왜 바꿨는지 명확히).
2. `git push origin main`을 시도한다.
   - 이 저장소가 로컬 인증 정보 없는 환경(Claude의 격리된 작업환경 등)에 연결돼 있으면 push가 실패할 수 있다 — 이 경우 사용자에게 "GitHub Desktop에서 Push branch 눌러달라"고 안내하고 마무리한다.
   - 인증 정보가 있는 환경(GitHub Desktop이 직접 연결된 로컬 폴더 등)이면 바로 push까지 완료한다.

## 브랜치 전략

기본적으로 main에 바로 작업한다(1인 프로젝트라 충돌 위험 낮음, 문서 작업 위주). 다만 아래 경우에는 Claude가 먼저 판단해서 "이번 건 브랜치 나눠서 할까요?"라고 사용자에게 먼저 물어보고 진행한다 — 판단은 Claude가 하되 실행 승인은 사용자에게 받는다.

- 실험적이거나 실패 가능성이 있는 큰 변경(예: 컴포넌트 대량 리팩토링, 라이브러리 교체)
- 기존에 잘 동작하던 코드/문서를 크게 뜯어고치는 작업
- 결과가 마음에 안 들면 되돌리고 싶어질 만한 작업

단순 문서 추가/수정, 작은 버그 수정, 체크리스트 갱신 같은 건 브랜치 없이 바로 main에 커밋한다.

## 알려진 이슈

- 이 레포 폴더가 로컬 GitHub Desktop과 동시에 열려 있으면, Claude 작업환경에서 git 명령 실행 시 `.git/index.lock`, `.git/HEAD.lock` 같은 잠금 파일이 이미 있다는 에러가 종종 발생한다. `rm`으로 안 지워지면(`Operation not permitted`) `mv`로 파일명을 바꿔서 치우면 정상 진행된다. 삭제가 아니라 이름 변경이 핵심.
- Claude의 격리된 작업환경에는 GitHub 인증 정보가 없어서 `git push`는 항상 실패한다(의도된 제약 — 토큰을 Claude가 직접 다루지 않기 위함). 커밋까지만 Claude가 하고, push는 사용자가 GitHub Desktop에서 직접 눌러야 한다.

## 이 프로젝트의 목적

MES(생산관리시스템) 제품의 UI 컴포넌트 라이브러리를 Figma 디자인 시스템 기준으로 구축해서, 이후 신규 화면을 AI 바이브코딩할 때 참고 컨텍스트로 쓰는 것. 담당자(이해빈)는 디자이너/기획자이며 코드를 직접 다루지 않는다 — Figma 추출, 토큰 실측, 코드 작성, 환경 구축, 배포까지 전 과정이 Claude의 실행 책임이다.

## 향후 스코프 (아직 일정 미확정)

기존에 이미 개발/배포된 MES 시스템이 있으며, 장기적으로는 이 신규 컴포넌트 라이브러리 기준으로 기존 화면을 리팩토링하는 것도 목표에 포함된다(현재 일정 미확정, 컴포넌트 라이브러리 완성이 선행 조건). 원자 단위 컴포넌트(버튼/뱃지/모달 등)는 화면별 매핑 불필요 — 교체 시 전 화면에 자동 반영됨. MES 특화 복합 패턴(화면 템플릿 등)만 화면 대응 기록이 의미 있음.
